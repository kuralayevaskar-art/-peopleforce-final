# Задача для Codex: доработка системы сотрудников DMUK

## Главная цель

Нужно доработать существующую систему сотрудников так, чтобы она поддерживала:

- редактирование карточек сотрудников администратором;
- вход сотрудников через Active Directory;
- ограничение доступа обычных пользователей только к своей странице и своей структуре;
- отдельную страницу регистрации новых пользователей;
- модерацию заявок администратором;
- автоматическое создание пользователя в Active Directory;
- запуск синхронизации Microsoft 365;
- создание пользователя в ZKT и загрузку фото для Face ID;
- отправку логина и временного пароля на личную почту;
- безопасную генерацию уникального временного пароля;
- журналирование действий, ошибок и входов.

---

## Важные правила для реализации

1. Не хардкодить пароли, токены, учетные данные AD, M365, ZKT или SMTP.
2. Все секреты брать из `.env`, переменных окружения или текущего механизма конфигурации проекта.
3. Не использовать один общий пароль для всех пользователей.
4. Для каждого нового пользователя генерировать уникальный временный пароль.
5. Пароль не должен храниться в базе данных в открытом виде.
6. Временный пароль должен быть отправлен пользователю только один раз.
7. При первом входе пользователь обязан сменить временный пароль.
8. Обычный пользователь не должен иметь доступ к данным других сотрудников.
9. Обычный пользователь не должен иметь возможность менять ФИО, корпоративную почту, роль, отдел и системные данные AD.
10. Все действия администратора, ошибки и входы пользователей должны записываться в журнал.

# 1. Роли пользователей

В системе должны быть минимум две роли:

## 1.1. Администратор

Администратор может:

- просматривать всех сотрудников;
- редактировать карточки сотрудников;
- просматривать заявки на регистрацию;
- одобрять заявки;
- отклонять заявки с указанием причины;
- создавать пользователей в AD через систему;
- запускать процесс синхронизации с Microsoft 365;
- создавать аккаунты в ZKT;
- загружать фото в ZKT;
- управлять отделами;
- просматривать журнал действий;
- просматривать ошибки создания и обновления пользователей.

## 1.2. Обычный пользователь

Обычный пользователь после входа через AD может:

- видеть только свою страницу;
- видеть только свою разрешенную структуру;
- редактировать только разрешенные поля своей карточки.

Обычный пользователь может изменять:

- фото;
- номер телефона;
- личную почту;
- дополнительную информацию, если она уже предусмотрена системой.

Обычный пользователь НЕ может изменять:

- ФИО;
- корпоративную почту;
- роль;
- отдел;
- должность, если она управляется администратором;
- системные данные AD;
- данные других сотрудников.

Данные сотрудника успешно обновлены.

# 4. Исправить вход через Active Directory

На странице входа есть кнопка:

```text
Войти / Login with AD
```

Сейчас вход через AD не работает. Нужно исправить.

## 4.1. Требования к авторизации

- вход через AD должен работать для всех сотрудников, которые есть в Active Directory;
- после успешного входа система должна определить пользователя по корпоративной учетной записи;
- администратор после входа получает доступ к панели администратора;
- обычный пользователь после входа получает доступ только к своему профилю и разрешенной структуре;
- обычный пользователь не должен видеть данные других сотрудников;
- пользователь не должен получать доступ к admin routes без роли администратора.

Пример корпоративного логина:

```text
a.kuralayev@dmuk.edu.kz
```

## 4.2. Если пользователь есть в AD, но нет в базе сайта

Если пользователь найден в AD, но его профиль отсутствует в базе сайта, нужно сделать один из вариантов в зависимости от архитектуры проекта:

Вариант A, предпочтительный:

- создать связанный профиль автоматически с минимальными данными из AD;
- назначить роль обычного пользователя;
- записать действие в журнал.

Вариант B:

- показать понятное сообщение.

Пример сообщения:

```text
Ваша учетная запись найдена в AD, но профиль в системе еще не создан. Обратитесь к администратору.
```

# 10. Уникальная ссылка на регистрацию

Регистрация должна работать по уникальной ссылке.

Требования к ссылке:

- ссылка уникальная;
- ссылка имеет срок действия, например 3–7 дней;
- ссылка используется только один раз;
- после отправки заявки ссылка становится недействительной;
- если срок действия истек, пользователь должен запросить новую ссылку у администратора;
- токен ссылки нельзя хранить в открытом виде, если в проекте уже есть механизм безопасного хранения токенов.

Пример сообщения при недействительной ссылке:

```text
Ссылка недействительна или срок ее действия истек. Обратитесь к администратору.
```

Пользователь должен загрузить фото для Face ID.

На странице регистрации нужно показать пример правильного фото.

Требования к фото:

- хорошее освещение;
- фото без других людей;
- лицо не закрыто;
- без темных очков;
- головной убор не закрывает лицо;
- фото четкое.

В UI рядом с загрузкой фото добавить блок:

```text
Пример правильного фото для Face ID
```

Также желательно добавить примеры неправильных фото:

- лицо сбоку;
- плохое освещение;
- лицо закрыто;
- фото размыто;
- на фото несколько человек.

Если в проекте нет готовой картинки-примера, создать простой placeholder/схему в текущем стиле интерфейса.

# 13. Фото удостоверения личности

Пользователь должен загрузить фото удостоверения личности.

- допустимые форматы: JPG, PNG, PDF;

# 14. Защита персональных данных

Система будет хранить личные данные, фото лица и фото удостоверения личности.

Требования:

- обычные пользователи не должны видеть документы других сотрудников;
- доступ к документам должен быть ограничен ролью администратора;
- файлы хранить в защищенном месте;
- прямые публичные ссылки на документы запрещены;
- все действия с документами записывать в журнал;
- удаление или изменение документов доступно только администратору;
- если хранение документов после проверки не требуется, предусмотреть возможность удаления после одобрения.

# 15. Панель администратора для одобрения заявок

В панели администратора нужно добавить раздел:

```text
Заявки на регистрацию
```

или

```text
Одобрение пользователей
```

Администратор должен видеть список заявок.

## 15.1. Статусы заявок

У заявки должны быть статусы:

- новая;
- на проверке;
- одобрена;
- отклонена;
- ошибка создания пользователя;
- пользователь создан.

## 15.2. Просмотр заявки

Администратор должен открыть заявку и проверить:

- ФИО;
- номер телефона;
- личную почту;
- фото удостоверения личности;
- фото для Face ID;
- корректность заполненных данных.

## 15.3. Одобрение заявки

Если данные корректные, администратор нажимает:

```text
Одобрить
```

После этого система запускает процесс создания пользователя.

## 15.4. Отклонение заявки

Если данные некорректные, администратор нажимает:

```text
Отклонить
```

При отклонении администратор должен указать причину.

Примеры причин:

- некорректное фото;
- не загружено удостоверение;
- ошибка в ФИО;
- неверный номер телефона;
- неверная личная почта;
- данные заполнены не полностью.

После отклонения пользователь должен получить письмо на личную почту с причиной отказа.

# 16. Проверка данных перед созданием пользователя

Перед созданием пользователя система должна проверить:

- заполнены ли все обязательные поля;
- корректно ли указана личная почта;
- корректно ли указан номер телефона;
- загружено ли фото для Face ID;

- нет ли пользователя с таким ФИО;

- нет ли пользователя с таким номером телефона;

- корректно ли сформирован логин.

Если найден дубль, система должна показать предупреждение и не создавать пользователя автоматически без подтверждения администратора.

Пример:

```text
Найден пользователь с таким ФИО. Проверьте данные перед созданием.
```

# 17. Формирование корпоративной почты

Корпоративная почта формируется автоматически по ФИО пользователя.

Формат:

```text
первая_буква_имени.фамилия@dmuk.edu.kz
```

Пример:

```text
ФИО: Куралаев Аскар Акатаевич
Почта: a.kuralayev@dmuk.edu.kz
```

Если такая почта уже существует, система должна предложить другой вариант:

```text
a.kuralayev1@dmuk.edu.kz
```

или

```text
askar.kuralayev@dmuk.edu.kz
```

Администратор должен видеть сформированную корпоративную почту до создания пользователя.

Важно:

- нужно корректно обработать кириллицу;
- фамилию и имя нужно транслитерировать латиницей;
- все буквы в email должны быть в нижнем регистре;
- пробелы и лишние символы удалить;
- проверить дубли в базе и AD.

# 18. Генерация уникального временного пароля

Запрещено использовать один общий пароль для всех новых пользователей.

Система должна автоматически генерировать уникальный временный пароль для каждого пользователя.

Требования к паролю:

- генерируется автоматически;
- уникальный для каждого пользователя;
- минимум 12 символов;
- содержит заглавные буквы;
- содержит строчные буквы;
- содержит цифры;
- содержит специальные символы;
- не совпадает с ФИО пользователя;
- не совпадает с телефоном пользователя;
- не совпадает с почтой пользователя;
- не хранится в базе в открытом виде.

Пример временного пароля:

```text
Dmuk@A7xP92qL
```

Важно для реализации:

- использовать криптографически стойкий генератор случайных значений;
- не логировать пароль;
- не показывать пароль администратору после отправки;
- передавать пароль только в момент создания AD-пользователя и отправки письма.

# 19. Смена пароля при первом входе

После создания пользователя в Active Directory нужно включить требование смены пароля при первом входе.

В AD должен быть установлен параметр:

```text
User must change password at next logon
```

Требования:

- пользователь обязан сменить временный пароль при первом входе;
- пока пароль не изменен, доступ к системе должен быть ограничен;
- статус пользователя должен показывать, что требуется смена пароля, если проект это поддерживает.

# 20. Создание пользователя в Active Directory

После одобрения заявки система автоматически создает пользователя в Active Directory.

В AD передаются:

- ФИО;
- корпоративная почта;
- логин;
- номер телефона;
- отдел;
- должность, если указана;
- личная почта, если такое поле используется;
- временный пароль;
- флаг обязательной смены пароля при первом входе.

Если пользователь успешно создан в AD, статус заявки меняется на:

```text
Пользователь создан в AD
```

Если создать пользователя не удалось:

- показать понятную ошибку администратору;
- записать ошибку в журнал;
- не продолжать процесс M365/ZKT, если AD-пользователь не создан.

---

# 21. Синхронизация с Microsoft 365

После создания пользователя в AD нужно запустить синхронизацию с Microsoft 365.

Команда:

```powershell
Start-ADSyncSyncCycle -PolicyType Delta
```

Требования:

- запускать команду только на разрешенном сервере/через безопасный backend-service;
- не запускать PowerShell из frontend;
- записать результат запуска в журнал;
- показать статус администратору.

Возможные статусы:

- синхронизация запущена;
- синхронизация выполнена успешно;
- синхронизация завершилась ошибкой.

Пример ошибки:

```text
Пользователь создан в AD, но синхронизация с Microsoft 365 не выполнена.
```

---

# 22. Создание пользователя в ZKT

После создания пользователя в AD и запуска синхронизации нужно создать пользователя в ZKT.

Требования:

- создание в ZKT выполнять примерно через 5 минут после создания в AD;
- лучше использовать очередь задач, scheduled job или background worker;
- не блокировать интерфейс администратора на 5 минут;
- передать в ZKT данные пользователя;
- загрузить фото для Face ID;
- записать результат в журнал.

В ZKT передаются:

- ФИО;
- корпоративная почта;
- номер телефона;
- фото для Face ID;
- ID пользователя, если используется;
- отдел, если используется в ZKT.

Если пользователь успешно создан в ZKT, статус меняется на:

```text
Пользователь создан в ZKT
```

Если ошибка:

```text
Пользователь создан в AD и Microsoft 365, но не создан в ZKT.
```

или

```text
Не удалось загрузить фото пользователя в ZKT.
```

---

# 23. Отправка письма пользователю

После успешного создания учетной записи система отправляет письмо на личную почту пользователя.

В письме должны быть:

- корпоративный логин;
- временный пароль;
- инструкция по первому входу;
- требование сменить пароль;
- предупреждение не передавать пароль другим людям.

Пример письма:

```text
Тема: Данные для входа в корпоративную учетную запись

Здравствуйте!

Ваша корпоративная учетная запись успешно создана.

Ваш логин:
a.kuralayev@dmuk.edu.kz

Ваш временный пароль:
Dmuk@A7xP92qL

При первом входе в систему необходимо обязательно сменить временный пароль.

Не передавайте логин и пароль другим лицам.

С уважением,
Администрация DMUK
```

Важное требование:

- временный пароль отправляется только один раз;
- после отправки пароль не должен повторно отображаться администратору;
- пароль нельзя писать в logs.

---

# 24. Рекомендуемое улучшение безопасности

Если архитектура проекта позволяет, лучше вместо отправки временного пароля отправлять пользователю одноразовую ссылку для установки пароля.

Рекомендуемый безопасный вариант:

1. пользователь получает письмо на личную почту;
2. в письме находится одноразовая ссылка;
3. пользователь переходит по ссылке;
4. пользователь сам задает пароль;
5. ссылка становится недействительной.

Если этот вариант сейчас сложно реализовать, допускается использовать уникальный временный пароль с обязательной сменой при первом входе.

# 29. Полный процесс регистрации пользователя

Процесс должен работать так:

1. Администратор отправляет пользователю уникальную ссылку на регистрацию.
2. Пользователь открывает ссылку.
3. Пользователь заполняет ФИО, телефон, личную почту.
4. Пользователь загружает фото удостоверения личности.
5. Пользователь загружает фото для Face ID.
6. Система проверяет обязательные поля.
7. Пользователь отправляет заявку.
8. Заявка появляется в панели администратора.
9. Администратор проверяет данные.
10. Если данные некорректные, администратор отклоняет заявку с причиной.
11. Если данные корректные, администратор нажимает «Одобрить».
12. Система проверяет пользователя на дубли.
13. Система формирует корпоративную почту.
14. Система генерирует уникальный временный пароль.
15. Система создает пользователя в Active Directory.
16. В AD включается требование смены пароля при первом входе.
17. Система запускает синхронизацию:

```powershell
Start-ADSyncSyncCycle -PolicyType Delta
```

18. Пользователь появляется в Microsoft 365.
19. Примерно через 5 минут система создает пользователя в ZKT.
20. Система загружает фото пользователя в ZKT.
21. Система отправляет логин и временный пароль на личную почту пользователя.
22. Все действия записываются в журнал.

# 30. Критерии приемки

Работа считается выполненной, если:

- кнопка «Редактировать» видна только администратору;
- администратор может изменить данные сотрудника;
- изменения сохраняются в системе и обновляются в AD;
- поле Search удалено с главной страницы;
- вход через AD работает;
- обычный пользователь видит только свою страницу;
- обычный пользователь может менять только разрешенные поля;
- обычный пользователь не может менять ФИО и корпоративную почту;
- ошибка создания пользователя исправлена;
- отделы выбираются из выпадающего списка;
- лишняя кнопка «Создать» убрана;
- страница регистрации работает по уникальной ссылке;
- все обязательные поля проверяются;
- без фото удостоверения и фото для Face ID заявку отправить нельзя;
- заявка появляется в панели администратора;
- администратор может одобрить заявку;
- администратор может отклонить заявку с причиной;
- после одобрения пользователь создается в AD;
- после создания запускается синхронизация с Microsoft 365;
- пользователь создается в ZKT;
- фото загружается в ZKT;
- пользователь получает письмо на личную почту;
- пароль генерируется уникальный для каждого пользователя;
- при первом входе пользователь обязан сменить пароль;
- все действия записываются в журнал;
- ошибки записываются в журнал;
- backend защищает права доступа, а не только UI;
- доступны тесты или ручные сценарии проверки.

# 32. Технические заметки для Codex

- Не удалять существующую бизнес-логику без необходимости.
- Все проверки прав должны быть на backend/API уровне.
- UI-проверки являются дополнительными, но не основными.
- Для фоновой задачи ZKT использовать существующий механизм очередей, cron, scheduler или background jobs, если он есть в проекте.
- Если очередей нет, создать минимальный безопасный сервисный слой и описать, как подключить реальный scheduler.
- Для AD, M365, ZKT сделать отдельные сервисы, а не писать интеграционную логику прямо в контроллере/компоненте.
- Для email использовать существующий mailer проекта, если он есть.
- Для файлов использовать существующее файловое хранилище проекта, если оно есть.
- Для audit logs использовать существующую таблицу/модель логов, если она есть; если нет — добавить новую.
- Не сохранять временный пароль в audit logs.
- Не показывать личные документы обычным пользователям.
